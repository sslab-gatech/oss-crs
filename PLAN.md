# OSS-CRS Plan

This document tracks the current state of OSS-CRS, upcoming features, and the long-term roadmap. It is intended to give contributors and users a clear view of what works today, what's coming next, and where help is needed.

---

## Current State (What's Working)

### CRS Compose (Orchestration Layer)

- [x] Three-phase lifecycle: `prepare` ‚Üí `build-target` ‚Üí `run`
- [x] `local` runtime environment via Docker Compose
- [x] `crs-compose.yaml` configuration with Pydantic-based validation
- [x] Per-CRS resource isolation (`cpuset`, `memory`, `llm_budget`)
- [x] CRS sourcing from **git URLs** (url + ref) and **local paths**
- [x] Docker Buildx Bake integration for the prepare phase
- [x] Multi-module CRS support (multiple containers per CRS with private networking)
- [x] Ensemble composition (run multiple CRSs in a single campaign)

### libCRS (CRS Communication Library)

- [x] `submit-build-output` ‚Äî rsync-based build artifact submission
- [x] `download-build-output` ‚Äî retrieve build artifacts during run phase
- [x] `skip-build-output` ‚Äî mark optional outputs as intentionally skipped
- [x] `register-submit-dir` ‚Äî background daemon with `watchdog` for batch submission (dedup by MD5)
- [x] `register-shared-dir` ‚Äî symlink-based inter-container file sharing within a CRS
- [x] `submit` ‚Äî manual single-file submission (`pov`, `seed`, `bug-candidate`)
- [x] `get-service-domain` ‚Äî DNS-verified service endpoint resolution

### oss-crs-infra (Shared Infrastructure)

- [x] **LiteLLM Proxy** ‚Äî unified LLM API endpoint, per-CRS API keys, budget enforcement, model routing
- [x] **PostgreSQL** ‚Äî LiteLLM state and budget tracking
- [x] **Storage** ‚Äî shared filesystem for seeds, PoVs, bug candidates, and patches
- [x] **litellm-key-gen** ‚Äî service that registers per-CRS keys and budgets at launch

### Examples & Registry

- [x] `crs-libfuzzer` ‚Äî lightweight libFuzzer-based CRS example
- [x] `atlantis-multilang-wo-concolic` ‚Äî LLM-powered multi-language CRS
- [x] Ensemble composition example (both CRSs running simultaneously)
- [x] CRS registry format (`registry/<name>/pkg.yaml`)

## Near-Term (Next Up)

These items are the highest-priority incomplete features, critical for making ensemble campaigns and the CRS lifecycle fully functional.

### Cross-CRS Data Sharing

| Item | Status | Notes |
|---|---|---|
| `register-fetch-dir` | Registered in CLI, **not yet implemented** | Enable CRSs to receive shared seeds/PoVs from other CRSs |
| `fetch` (pov, seed, bug-candidate) | Registered in CLI, **not yet implemented** | Core primitive for ensemble value |

## Known Limitations

1. **No cross-CRS runtime data sharing** ‚Äî `fetch` / `register-fetch-dir` are stubs, so ensembled CRSs cannot share seeds or PoVs during a campaign.
2. **No crash deduplication** ‚Äî multiple CRSs in an ensemble may report the same bug independently.
3. **No campaign monitoring** ‚Äî no WebUI or dashboard to observe progress in real-time.
4. **No patching pipeline** ‚Äî bug-fixing CRS types cannot yet execute `apply-patch-build`.
5. **Single deployment target** ‚Äî only `local` (Docker Compose) is functional; Azure is config-supported but not implemented.
6. **Standalone CRS CLI not functional** ‚Äî individual CRS management (outside compose) is not yet available.

## How to Contribute

See [CONTRIBUTING.md](CONTRIBUTING.md) for contribution guidelines. Areas where help is especially welcome:

- **Infrastructure services** ‚Äî designing and implementing seed dedup, PoV dedup, and WebUI
- **Cross-CRS sharing** ‚Äî implementing `fetch` / `register-fetch-dir` in libCRS
- **Azure backend** ‚Äî `AzureCRSUtils` and Azure deployment support
- **New CRS integrations** ‚Äî packaging existing bug-finding or bug-fixing tools as OSS-CRS-compatible CRSs
- **Validation & robustness** ‚Äî tightening config validation and error handling across the codebase
